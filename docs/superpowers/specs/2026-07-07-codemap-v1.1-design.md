# CodeMap v1.1 — Multi-Language Analysis + Home Screen v2 — Design

**Date:** 2026-07-07 · **Status:** Scope approved by user (universal + top-8 deep; all four home-screen elements)
**Baseline:** v1 spec (`2026-07-07-codemap-design.md`) — everything below is additive.

## A. Multi-language analysis (universal ingest, top-8 deep)

**Goal:** point CodeMap at a repo in any language. Every source file becomes a node with an LLM explanation and is chat/RAG-searchable. Deep languages additionally get real import edges and class/function symbols.

**Deep languages (import edges + symbols):** Python (existing `ast` path, unchanged), JavaScript, TypeScript (+JSX/TSX), Go, Java, Rust, C#, C/C++, Ruby, PHP.
**Universal tier:** any other extension with a tree-sitter grammar in `tree-sitter-language-pack` (Kotlin, Swift, Scala, Lua, …) gets nodes + explanations + chunks; symbols where a generic query works, no import edges. Unknown-but-texty code extensions get raw-fallback nodes.

### Backend changes

1. **Deps:** `tree-sitter>=0.23`, `tree-sitter-language-pack` (prebuilt wheels, no compilers).
2. **`analyzer/languages.py` (new):** registry `EXT_TO_LANGUAGE`; per-deep-language spec: tree-sitter language name, import query (tree-sitter query source), symbol query (functions/classes/methods with doc-comment capture where the grammar supports it), and a module-name strategy (how a file path becomes a dotted/importable name for that language).
3. **`analyzer/parser.py`:** `parse_module` dispatches: `.py` → existing ast path (untouched, keeps all current tests green); deep language → tree-sitter parse → imports + ClassInfo/FunctionInfo from query captures; universal tier → tree-sitter parse for symbol counts if grammar exists, else raw fallback (`status="ok"`, empty symbols). Parse failures anywhere → `status="parse_error"` (never fatal). `ModuleInfo` gains `language: str` (additive).
4. **Import resolution per language** (all resolve to repo-relative file paths; unresolvable imports drop, as today):
   - JS/TS: relative specifiers (`./x`, `../y`) resolved with extension/index probing (`.ts .tsx .js .jsx /index.*`); path-mapped/bare specifiers ignored in v1.1.
   - Go: package-dir imports resolved against `go.mod` module path prefix; same-module only.
   - Java/C#: package/namespace dotted names mapped to directory paths (longest-prefix, reusing v1 `_resolve_target`).
   - Rust: `mod x;` sibling/`x/mod.rs` resolution + `use crate::…` paths.
   - C/C++: local `#include "…"` relative resolution (angle-bracket includes ignored).
   - Ruby/PHP: `require_relative`/`use` with best-effort path mapping.
5. **Skeleton builder:** language-aware — signatures + doc comments from tree-sitter captures under the same 16k-char budget; raw-fallback files use the first 16k chars of source with a header noting the language.
6. **Chunker:** symbol-boundary chunks where symbols exist; else one whole-file chunk (embed-side truncation already bounds memory).
7. **Fixtures/tests:** new polyglot fixture (`demo_polyglot/`: small TS+JS, Go with go.mod, Java, Rust set with known cross-file imports); per-language import-resolution tests; existing Python fixture and all 42 tests untouched. LLM/embeddings still fully mocked.
8. **Frontend:** `GraphNode.language` in types; designation plate shows the language as part of the readout row; package coloring unchanged.

**Non-goals (v1.1):** cross-repo/vendored deps, bare-specifier JS resolution via node_modules (excluded dir anyway), build-system awareness beyond `go.mod`, per-language LOC semantics.

## B. Home screen v2

1. **Recents (backend):** on each successful pipeline run, upsert `{repo_path, name, analyzed_at, modules, packages, languages}` into a server-local registry (`~/.codemap/recents.json`, capped at 12, most-recent first; in Docker this lives in the container volume). New route `GET /recent` returns the list; entries whose paths no longer exist are filtered out. Clicking a recent card POSTs `/analyze` — v1.1's warm-start makes it near-instant.
2. **Browse for folder (server-side):** the browser File System Access API cannot yield absolute paths, so Browse is a server-side directory navigator: `GET /fs?path=<dir>` returns `{parent, dirs:[{name, path}]}` rooted at the backend's filesystem (Docker: `/repos`). UI: popover panel from a "Browse" button beside the path input — navigate, pick, path fills the input. Read-only; directories only; hidden/system dirs filtered.
3. **How-it-works strip:** three quiet instrument-styled steps under the input — "Parse · every file becomes a star", "Explain · the model describes each module", "Explore · ask questions, fly to answers". Sentence-case copy, no icons-as-decoration beyond three small lucide glyphs.
4. **Live teaser:** ambient constellation behind the hero — pure CSS/canvas (drifting star field + occasional connecting line), `motion-safe` gated, no three.js on the home screen (keeps first paint instant).

**Route/API summary (additive):** `GET /recent`, `GET /fs?path=`. Both localhost-scoped like everything else; `/fs` never lists files, only directories, and rejects paths that aren't directories.

## Success criteria

- Analyze a real TypeScript repo (e.g. zustand) end-to-end in Docker: nodes for all source files, import edges between TS modules, explanations, chat citations fly the camera.
- Analyze a mixed repo: Python and non-Python files coexist in one graph.
- Home screen: previously analyzed repos appear as cards and re-open warm; Browse navigates the server filesystem; all v1 gates stay green (backend suite grows with per-language tests; mypy/ruff/tsc/eslint clean).
