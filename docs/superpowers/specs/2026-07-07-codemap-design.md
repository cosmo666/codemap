# CodeMap — "Explain This Codebase" 3D Visualizer — Design

**Date:** 2026-07-07
**Status:** Approved pending user review
**Working name:** CodeMap (rename freely)

## 1. Overview

CodeMap is a local web application that points at a Python repository and produces a cinematic, interactive 3D map of its architecture. Every module becomes a glowing node in a three.js scene; imports become directional particle streams; each node carries a plain-English explanation generated by an LLM. A RAG-powered chat answers questions about the codebase ("where is auth handled?") with structured citations that fly the camera to the relevant modules.

**Dual purpose:**

1. **Real value:** dramatically faster onboarding to unfamiliar Python codebases.
2. **Learning vehicle:** teaches AST-based code analysis, map-reduce LLM summarization, embeddings + vector search, RAG with citations, SSE streaming, and production engineering practices.

**Inspiration:** architecture and polish level of [obsidian-ai](https://github.com/sup3rus3r/obsidian-ai) (FastAPI backend, React frontend, SSE streaming, FAISS RAG).

## 2. Decisions Locked During Brainstorming

| Decision | Choice |
| --- | --- |
| Core experience | Explore + chat (3D map is the entry point; RAG chat over the code) |
| Language support (v1) | Python only, via the built-in `ast` module |
| Repo scale (v1) | Small–medium: up to ~300 Python files, flat (non-hierarchical) graph |
| LLM | GPT-4.1 mini via user-provided OpenAI-compatible endpoint |
| Embeddings | Local (`fastembed`, ONNX/CPU) — independent of LLM endpoint quota |
| Architecture | FastAPI backend + React (Vite) frontend with `react-force-graph-3d` |
| Visual bar | Cinematic: bloom, particle edges, camera choreography, dark constellation aesthetic |
| Quality bar | Enterprise-grade engineering rigor now; enterprise features as explicit v2 roadmap |

## 4. Backend Components

### 4.1 Analyzer (pure, deterministic, no LLM)

- Walks the target repo; respects `.gitignore`-style exclusions (venv, `__pycache__`, hidden dirs).
- Parses each `.py` file with `ast`; extracts imports, classes, functions, docstrings, line counts.
- Builds a directed import graph with `networkx`: **nodes** = modules, **edges** = "imports from", **clusters** = packages (folders).
- Unparseable files are not fatal: recorded as nodes with `"status": "parse_error"`.
- Computes per-node metrics used by the frontend: import centrality (node size), package (color), LOC.

### 4.2 Explainer (map-reduce LLM stage)

- **Map:** per module, send a skeleton — all class/function signatures, all docstrings, and module-level code, truncated to a fixed token budget (default ~4k tokens per module) — to GPT-4.1 mini → JSON explanation: what it does, its role in the system, key entities. Runs concurrently under a rate limiter (default 5 parallel, configurable).
- **Reduce:** module explanations roll up into per-package summaries, then one repo-level overview.
- **Cache:** keyed by file content hash. Re-runs only re-explain changed files. Failed explanations are marked `"unexplained"` and retried on next run.

### 4.3 Indexer

- Chunks at function/class boundaries from the AST (no line-window splitting). Module-level code becomes its own chunk.
- Embeds chunks with `fastembed` (local, CPU, no API key); stores in FAISS.
- Also embeds the prose module explanations, enabling hybrid retrieval (code chunks + summaries).
- Swappable: if the user's endpoint later exposes an embedding model, the embedder is behind an interface.

### 4.4 API (FastAPI)

| Endpoint | Behavior |
| --- | --- |
| `POST /analyze` | Body: `{repo_path}`. Starts pipeline as background task; returns `analysis_id`. |
| `GET /analyze/{id}/events` | SSE stream of progress events (`parsing`, `explaining 47/183`, `indexing`, `done`, `error`). |
| `GET /graph` | Full graph JSON: nodes (with explanations, metrics, status), edges, packages, repo overview. |
| `GET /module/{path}` | Per-module detail: explanation, classes/functions, dependents, dependencies. |
| `POST /chat` | Body: `{question, history}`. SSE stream of answer tokens + structured citation events. |

### 4.5 Persistence (deliberately simple)

`.codemap/` data folder per analyzed repo: `graph.json`, `explanations.json`, FAISS index file, `hash_cache.json`. No database in v1 — every pipeline artifact is human-inspectable.

## 5. Chat / RAG Design

1. Question embedded locally → FAISS top-k over **both** code chunks and module summaries (hybrid retrieval).
2. Retrieved context + system prompt sent to GPT-4.1 mini. The prompt **requires** structured citations of module paths (`[cite: app/auth/session.py]`).
3. Answer streams over SSE; citations are parsed server-side and emitted as structured events alongside tokens.
4. If no retrieved chunk clears a cosine-similarity threshold (configurable, default 0.35), the context block is marked as weak and the model is instructed to say it cannot find relevant code rather than hallucinate.

## 6. Frontend Design

**Stack:** React 19 + Vite + TypeScript, `react-force-graph-3d` (three.js under the hood, raw scene access), Zustand (single store: graph data, selection, chat history, analysis progress). No Next.js — no routing/SSR needs.

**The cinematic layer:**

- **Aesthetic:** dark space background, subtle starfield, emissive nodes with UnrealBloomPass glow — the codebase reads as a constellation.
- **Nodes:** sized by import centrality, colored by package; `parse_error` nodes render dim red.
- **Edges:** directional particle streams showing dependency flow.
- **Camera choreography:** slow orbital establishing shot on load; click a node → camera flies to it, its dependency neighborhood stays lit, the rest dims; side panel slides in with the explanation.
- **Chat drives the camera:** cited nodes pulse as the answer streams; clicking a citation flies the camera there with the subgraph highlighted.
- **Analyze sequence is a scene:** as SSE progress arrives, nodes materialize package by package, edges shoot between them, the layout settles — with a live status ticker.

## 7. Error Handling

- Pipeline never dies on one bad file (skip + mark, visible in the map).
- LLM calls: retry with exponential backoff on rate limits/transient errors; concurrency cap; per-module failure marks `"unexplained"` and continues.
- Analysis runs as a background task; SSE client disconnects do not kill it.
- Chat with empty retrieval → honest "not found" behavior, enforced by prompt.
- API errors are structured (`{error, detail, module?}`) and surfaced in the UI, not swallowed.

## 8. Testing

- **Fixture repo** (`backend/tests/fixtures/demo_repo/`): ~10 Python files with known imports, a class hierarchy, and one deliberately broken file.
- **Analyzer unit tests:** exact assertions on node count, edge list, chunk boundaries, parse-error handling.
- **Explainer/chat tests:** mocked LLM client (no network, no cost); verifies map-reduce flow, caching, retry behavior, citation parsing.
- **One integration test:** full pipeline on the fixture repo with mocked LLM.
- **Frontend:** type-checked and linted; the 3D scene is verified visually (unit-testing three.js scenes has poor cost/value).

## 9. Engineering Rigor (the "enterprise grade" bar)

- **Python:** full type hints, `mypy --strict`, `ruff` (lint + format), `pydantic-settings` config, structured logging (`structlog`), `pyproject.toml` with pinned deps.
- **TypeScript:** strict mode, ESLint + Prettier.
- **CI:** GitHub Actions — lint, type-check, and test both halves on every push.
- **Docker:** `docker-compose up` brings up backend + frontend.
- **Docs:** README with architecture diagram, quickstart, and screenshots; FastAPI auto-generated OpenAPI docs.

## 10. Configuration

`backend/.env` (validated by pydantic-settings at startup, fail-fast with a clear message):

```ini
LLM_BASE_URL=...        # user's OpenAI-compatible endpoint
LLM_API_KEY=...
LLM_MODEL=gpt-4.1-mini
MAX_CONCURRENCY=5
```

The target repo path is runtime input (entered in the UI), not configuration.

## 11. Out of Scope for v1 / v2 Roadmap

**Out of scope for v1:** languages beyond Python; repos beyond ~300 files; file-watching / live re-analysis; multi-repo; auth.

**v2 roadmap (architecture leaves room, no v1 work):**

1. **Auth & API keys** — the API layer is already centralized; add middleware.
2. **Multi-repo workspaces** — persistence is already per-repo (`.codemap/` folders); add a workspace index.
3. **Job queue** — `POST /analyze` already returns an `analysis_id` and runs in the background; swap the in-process task for a queue (e.g. arq/Redis) without API changes.
4. **Hierarchical zoom for large repos** — packages-first view, zoom into modules.
5. **More languages** — tree-sitter behind the same Analyzer interface.

## 12. Success Criteria

- Point it at a real ~100–300-file Python repo (e.g. FastAPI's own source): analysis completes, the map renders at 60fps, every module has a sensible explanation.
- Ask "where does X happen?" for five known-answer questions: citations point at the right modules, camera flies correctly.
- A stranger can `docker-compose up`, open the browser, and understand a repo they've never seen.
- CI is green: lint, mypy, tests.
